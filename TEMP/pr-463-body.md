## Проблема

`addSignature()` в `core/security/key-management.ts` (строки 1439–1472) — это шлюз сбора подписей для пороговых/мультиподписных запросов на перевод средств. Метод проверял каждую входящую подпись и сохранял булево `verified`, но при сравнении с порогом считал **сырую длину массива**, а не число подтверждённых подписей:

```ts
const verified = await this.storage.verify(signature.publicKey, request.message, signature.signature);
request.collectedSignatures.push({ ...signature, verified });

if (request.collectedSignatures.length >= request.requiredSignatures) { // <-- баг: считает все строки
  request.status = 'ready_to_broadcast';
}
```

Поэтому запрос мог дойти до `ready_to_broadcast` с **невалидными** подписями: атакующий (или сбойный подписант), отправивший мусор, либо один и тот же подписант, отправивший подпись несколько раз под одним публичным ключом, набирал любой кворум N-of-N без валидных подписей. Контроль выглядел как пороговый, но фактически порог можно было обойти — для операций, двигающих средства, это критично.

## Решение

### `core/security/key-management.ts`

- **Кворум — только по проверенным подписям.** Порог `requiredSignatures` сравнивается с `collectedSignatures.filter((s) => s.verified).length`, а не с длиной массива. Неподтверждённые подписи хранятся (для аудита/трассировки), но в кворум не входят.
- **Один публичный ключ — один слот.** Перед учётом подписи метод ищет существующий слот по `publicKey`:
  - если слот уже занят **подтверждённой** подписью — повторная отправка по этому ключу отклоняется (`throw`), и понизить подтверждённый слот до неподтверждённого нельзя;
  - если слот хранит **неподтверждённую** попытку — она заменяется новой отправкой по тому же ключу (так junk-предсабмит под известным публичным ключом не блокирует реального подписанта).

  Поскольку каждый ключ держит максимум один слот, число подтверждённых подписей равно числу уникальных подтверждённых подписантов — один подписант не закрывает мультиподписный кворум в одиночку.
- Метод снабжён развёрнутым JSDoc с описанием инварианта и ссылкой на LOGIC-24 / issue #434.

### `tests/security/security.test.ts`

Добавлен блок из 4 регрессионных тестов `Threshold signature verification (LOGIC-24)` (для запроса 2-of-N на базе MPC, где `requiredSignatures === 2`):

- учитывает только проверенные подписи: одна валидная + одна невалидная подпись оставляют запрос в `collecting_signatures`, и только второй **валидной** подписи он переходит в `ready_to_broadcast`;
- две невалидные подписи **никогда** не дают `ready_to_broadcast` (кворум проверенных = 0);
- повторная подпись по уже подтверждённому публичному ключу отклоняется, длина коллекции остаётся 1;
- реальный подписант **заменяет** неподтверждённый предсабмит атакующего по тому же ключу (один слот на ключ, в итоге подтверждён).

Тесты обращаются к storage-бэкенду (`SoftwareKeyStorage`) для генерации настоящих подписей, поскольку `SecureKeyManager` не выдаёт приватные ключи.

## Критерии приёмки

- [x] Переход в `ready_to_broadcast` считает только подписи с `verified === true`.
- [x] Дублирующиеся публичные ключи не могут занять больше одного слота подписи.
- [x] Регрессионный тест: запрос с `requiredSignatures = 2` и одной валидной + одной невалидной подписью остаётся в `collecting_signatures`, переходя в `ready_to_broadcast` только после двух валидных подписей.

## Как воспроизвести

Standalone-скрипт сравнивает багованную и исправленную арифметику для запроса 2-of-N (`requiredSignatures: 2`):

```
node experiments/logic-review-v2/repro-LOGIC-24-threshold-counts-unverified.mjs
```

```
after 1 valid + 1 invalid signature:
  [buggy]  count rows      = 2 verified-or-not -> ready_to_broadcast
  [fixed]  count verified  = 1                 -> collecting_signatures
...
BUG REPRODUCED: the length-based gate accepts unverified/duplicate signatures; the verified-distinct gate does not.
```

(скрипт завершается с кодом `1`, когда баг воспроизведён)
