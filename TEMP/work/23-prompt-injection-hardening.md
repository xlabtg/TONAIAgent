# Task: Harden Prompt-Injection Detection Beyond Regex

**Priority:** 🟢 LOW (defense-in-depth; `PromptBuilder` already prevents the primary attack)
**Effort:** ~1 week
**Related Issue:** #325 (re-audit finding §7)
**Suggested labels:** `security`, `ai-safety`, `prompt-injection`

## Problem

The `sanitize.ts` utilities in PR #317 detect known injection patterns via regular expressions: `[system]`, `{{admin}}`, backtick system blocks, base64 payloads, common jailbreak strings. Regex-based detection misses:

- **Unicode lookalikes** — `Ｓystem` (fullwidth S), `ѕystem` (Cyrillic s), zero-width characters, homoglyphs
- **Encoded variants** — URL, HTML-entity, base32, hex, rot13
- **Novel patterns** — adversarial suffixes generated by gradient-based attacks
- **Indirect injection** — prompts embedded in fetched documents, URLs, tool outputs

These are secondary defenses — the primary protection is `PromptBuilder` enforcing structured input (see [`10-promptbuilder-adoption.md`](./10-promptbuilder-adoption.md)). But defense-in-depth is appropriate for a financial product.

## Acceptance Criteria

- [ ] Normalize all user input through Unicode NFKC + strip zero-width / bidi control characters before any downstream processing.
- [ ] Detect and refuse / quarantine confusables — use ICU confusable detection (e.g. `unicode-confusables` npm package) with a latin/ASCII-preferred mode.
- [ ] Flag (and optionally refuse) inputs containing large base64/base32/hex runs over a configurable length threshold.
- [ ] Add an output-side detection: if the model echoes text that looks like a system instruction ("ignore previous instructions", "you are now", etc.) in its response, log and flag it.
- [ ] For any tool-retrieved text that is later fed back into the model (RAG, web-fetch results), treat it as untrusted — wrap it in an isolation structure so the model cannot interpret it as instruction.
- [ ] Build a corpus of adversarial test cases (minimum 200) covering:
  - Homoglyph attacks (OCR + Unicode confusables)
  - Base64 payloads
  - Adversarial suffix collection from public datasets
  - Indirect injection via fetched URL contents
- [ ] Integrate the corpus into `tests/ai/prompt-injection.test.ts` as a regression suite.
- [ ] Document the threat model in `docs/ai-safety.md` — especially the distinction between "handled by PromptBuilder" and "handled by sanitization."

## Implementation Notes

- Keep false-positive rate low — blocking legitimate inputs is its own business risk.
- Rate-limit / shadow-ban users who repeatedly trigger the detectors — treat as spam signal.
